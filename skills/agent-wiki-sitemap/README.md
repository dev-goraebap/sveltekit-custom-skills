# agent-wiki-sitemap

PRD나 기획서를 입력하면 사이트맵(화면 구조도)을 Mermaid `.mmd` 파일로 생성하는 스킬이다. **agent-wiki**로 구성된 위키 환경이 전제 조건이며, 결과물은 `.wiki/design/`에 저장된다.

## 왜 만들었나

기능 요구사항이 정리되면 자연스럽게 "그래서 화면이 몇 개야?"라는 질문이 따라온다. 사이트맵은 그 질문에 대한 답이다. 어떤 화면이 있고, 어떻게 묶이며, 사용자가 어떤 경로로 접근하는지를 한눈에 보여준다.

이걸 수작업으로 그리면 PRD가 바뀔 때마다 다시 그려야 하는데, 텍스트 기반의 Mermaid flowchart로 만들면 에이전트에게 "여기에 화면 하나 추가해줘"라고 말하는 것만으로 수정이 끝난다. 위키에 PRD가 이미 있다면 입력도 따로 준비할 필요 없이 바로 생성할 수 있다.

## 고려할 점

- `.wiki` 심볼릭 링크가 없으면 동작하지 않는다. `agent-wiki` 스킬로 위키 환경을 먼저 구성해야 한다.

- 3단계 계층(대메뉴 → 중메뉴 → 기능 화면)을 기본으로 한다. 서비스 규모가 작으면 2단계로도 충분하고, 반대로 복잡한 경우에는 도메인별로 파일을 나누는 것이 낫다.

- 사이트맵은 설계 초기에 가장 유용하다. MVP가 어느 정도 완성된 뒤에는 소스코드의 라우팅 구조가 사실상 사이트맵 역할을 하므로, 위키에 상시 로드해둘 필요는 줄어든다.

## 워크플로우

### 사이트맵 생성

`사이트맵 만들어줘`와 같은 프롬프트를 요청한다. 에이전트가 입력(PRD, 기획서, 요구사항 등)을 분석하여 화면 계층을 도출하고, `.wiki/design/`에 `.mmd` 파일을 생성한다. 대메뉴는 subgraph, 중메뉴는 stadium 노드, 기능 화면은 rect 노드로 표현된다.

### 리뷰와 수정

VS Code의 Mermaid 프리뷰 확장으로 결과를 확인하고, 도메인 추가·삭제, 화면 재배치, 부연 설명 추가 등을 자연어로 요청하면 반영된다. 3-클릭 규칙(루트에서 모든 화면까지 3단계 이내)과 중복 여부도 함께 검증한다.
