# agent-wiki-tasks

`.wiki/tasks/`에 작업 지시서를 작성하고, Git 브랜치 전략에 따라 계획부터 완료까지 작업의 전체 라이프사이클을 관리하는 스킬이다. **agent-wiki**로 구성된 위키 환경이 전제 조건이다.

## 왜 만들었나

바이브 코딩을 하다 보면 "아까 뭐 하고 있었지?"라는 순간이 자주 온다. 컨텍스트가 날아갈 때마다 소스코드를 처음부터 훑어야 하고, 어떤 작업이 진행 중이었는지 기억에 의존하게 된다. Git 로그를 뒤져보면 힌트는 있지만, 커밋 메시지만으로 작업의 의도와 범위를 복원하기엔 부족하다.

그래서 작업 지시서라는 개념을 도입했다. 작업을 시작하기 전에 위키에 "무엇을, 왜 하는지"를 기록해두면, 나중에 새 컨텍스트에서 돌아왔을 때 에이전트가 그 파일을 읽는 것만으로 상황을 파악할 수 있다. 작업 결과는 Git 커밋과 태그에 위임하고, 지시서에는 의도와 참고 자료만 남긴다. 문서와 코드의 역할을 명확히 분리하는 것이다.

브랜치 전략도 함께 묶었다. 태스크 파일을 만들면 자동으로 작업 브랜치가 생기고, 작업이 끝나면 develop에 머지하고 버전 태그를 찍는다. 이 흐름을 매번 수동으로 하는 대신, "작업 완료"라는 한마디로 처리할 수 있게 하고 싶었다.

## 고려할 점

- `.wiki` 심볼릭 링크가 없으면 동작하지 않는다. `agent-wiki` 스킬로 위키 환경을 먼저 구성해야 한다.

- `develop` 브랜치 기반의 Git Flow를 전제한다. main에서 직접 작업하는 프로젝트에는 맞지 않을 수 있다. develop이 없으면 생성 여부를 확인한다.

- 태스크 파일은 scope 폴더로 프로젝트별 분리된다. 하나의 위키를 여러 프로젝트가 공유해도 태스크가 섞이지 않는다.

- 상태 파악 시 소스코드를 전체 탐색하지 않는다. 태스크 파일과 Git 메타데이터(branch, log, status, tag)만으로 현재 상황을 복원한다. 토큰 낭비를 줄이기 위한 의도적인 제약이다.

## 워크플로우

### 작업 내역 확인

새 대화에서 "어디까지 했지", "작업 현황"과 같은 프롬프트를 요청한다. 에이전트가 scope 폴더의 태스크 파일 목록과 Git 상태를 수집하여 현재 진행 중인 작업, 마지막 완료 버전, 커밋되지 않은 변경사항을 요약해준다. 소스코드를 읽지 않고도 작업 맥락을 빠르게 복원할 수 있다.

### 작업 계획

"구현해줘", "기능 추가해줘"와 같은 프롬프트를 요청한다. 에이전트가 요청을 분석하여 작업 타입과 버전을 결정하고, `.wiki/tasks/(scope)/`에 태스크 파일을 생성한 뒤, develop에서 작업 브랜치를 분기한다. 이후 지시 내용에 따라 구현을 진행한다.

### 작업 완료

"작업 완료"와 같은 프롬프트를 요청한다. 에이전트가 변경사항을 커밋하고, 사용자에게 승인을 받은 뒤 develop에 머지한다. annotated 태그를 생성하고 태스크 상태를 `done`으로 변경한 뒤, 위키 저장소에도 태스크 파일 변경을 커밋한다.
