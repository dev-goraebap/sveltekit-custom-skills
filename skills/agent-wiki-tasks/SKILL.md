# agent-wiki-tasks

`.wiki/tasks/` 에 작업 지시서를 작성하고, Git 브랜치 전략에 따라 작업의 전체 라이프사이클(계획 → 구현 → 완료)을 관리하는 스킬이다. `agent-wiki` 스킬로 구성된 위키 환경이 전제 조건이다.

태스크 파일 형식, 버전 결정 기준, 브랜치 전략 등 상세 규칙은 `references/conventions.md`를 참고한다.

## 가드레일

이 스킬의 모든 워크플로우는 아래 검증을 먼저 통과해야 한다. 하나라도 실패하면 안내 메시지를 출력하고 중단한다.

| # | 검증 | 실패 시 |
|---|------|---------|
| 1 | CWD가 Git 저장소임 | "Git 저장소에서 실행해주세요" |
| 2 | CWD 루트에 `.wiki` 심볼릭 링크가 존재하고 정상 동작 | "`agent-wiki` 스킬로 위키 환경을 먼저 구성해주세요. `위키 저장소 만들어줘`라고 요청하면 됩니다" |
| 3 | `.wiki` resolve 경로에 `.git`이 존재 | "위키 저장소가 Git으로 초기화되지 않았습니다" |
| 4 | `develop` 브랜치 존재 | "develop 브랜치가 없습니다. 현재 브랜치에서 생성할까요?" — 동의 시 `git checkout -b develop` |

## 워크플로우

### 0. 작업 내역 확인

새로운 컨텍스트에서 기존 작업 상태를 파악할 때 작동한다. "어디까지 했지", "작업 현황", "이어서 작업" 등의 요청이거나, 이미 작업 브랜치에 있는 상태에서 작업 요청이 들어온 경우에도 먼저 이 워크플로우를 실행한다.

1. **가드레일 검증**
2. **태스크 파일 목록 확인** — `ls .wiki/tasks/(scope)/`로 전체 태스크 파일명을 수집한다
3. **진행 중 태스크 읽기** — `in-progress` 상태인 태스크 파일이 있으면 내용을 읽는다
4. **Git 메타데이터 수집** — 아래 4개 명령만 실행한다:
   - `git branch` — 현재 브랜치와 작업 브랜치 목록
   - `git log --oneline -10` — 현재 브랜치의 최근 커밋
   - `git status --short` — 커밋되지 않은 변경사항 유무
   - `git tag --sort=-version:refname | head -5` — 최근 완료 태그
5. **상태 요약** — 수집한 정보를 종합하여 보여준다:
   - 진행 중인 작업 (태스크 파일 + 브랜치)
   - 마지막으로 완료된 작업 (최근 태그)
   - 커밋되지 않은 변경사항 유무
6. **다음 행동 제안** — 이어서 작업할지, 새 작업을 시작할지 사용자에게 확인한다

#### 금지 행위

이 워크플로우에서 상태 파악을 위해 아래 행위를 해서는 안 된다. 태스크 파일과 Git 메타데이터만으로 충분하다.

- 소스코드 파일 내용 읽기, 디렉토리 재귀 탐색
- `git diff`로 전체 변경사항 덤프 (변경 파일 목록은 `git status --short`로 충분)
- 프로젝트 구조 파악을 위한 glob/grep

### 1. 작업 계획

사용자가 소스코드 작업을 요청하거나 플랜 모드에 진입하여 소스코드를 다루려는 맥락일 때 작동한다.

1. **가드레일 검증**
2. **scope 폴더 확인** — `.wiki/tasks/(프로젝트명)/`이 있는지 확인하고, 없으면 생성한다. scope가 무엇인지는 `references/conventions.md`를 참고한다
3. **사용자 요청 분석** — 작업 타입(feat/fix/refactor/chore/docs)을 판단한다. 판단이 어려우면 사용자에게 확인한다
4. **버전 결정** — scope 폴더의 기존 태스크 파일들에서 다음 버전을 산출한다. 규칙은 `references/conventions.md`를 참고한다
5. **태스크 파일 작성** — 사용자의 지시 내용을 정리하여 기록하고, 상태를 `todo`로 설정한다
6. **작업 브랜치 생성** — `git checkout -b {타입}/{영문작업명} develop`
7. **태스크 상태를 `in-progress`로 변경**
8. **작업 수행** — 사용자의 지시에 따라 구현한다. 추가 수정 요청이 오면 계속 반영한다

### 2. 작업 완료

사용자가 작업 마무리를 요청할 때 작동한다.

1. **가드레일 검증**
2. **작업 브랜치 확인** — 현재 브랜치가 작업 브랜치 접두사(`feat/`, `fix/`, `refactor/`, `chore/`, `docs/`)를 가지는지 확인한다. `main`이나 `develop`이면 안내 후 중단
3. **진행 중인 태스크 파일 탐색** — `.wiki/tasks/(scope)/`에서 상태가 `in-progress`인 파일을 찾는다. 여러 개이면 현재 브랜치명과 매칭되는 것을 선택한다
4. **변경사항 커밋** — `git status` 확인 → 커밋 메시지 작성 → 커밋
5. **승인 요청** — 작업 요약을 보여주고 develop 머지 진행 여부를 사용자에게 확인한다
6. **develop 머지** — `git checkout develop && git merge --no-ff {브랜치명}`
7. **annotated 태그 생성** — `git tag -a v0.2.0 -m "feat: 디자인 시스템 구축"`
8. **태스크 상태를 `done`으로 변경**
9. **위키 커밋** — `.wiki` 원본 저장소에서 태스크 파일 변경사항을 커밋한다
10. **결과 안내** — 머지된 브랜치, 생성된 태그, 태스크 상태 변경을 요약
