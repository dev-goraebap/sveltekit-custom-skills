# 규칙 레퍼런스

## scope

scope는 하나의 위키 저장소 안에서 프로젝트별 태스크를 분리하기 위한 폴더명이다. 위키 저장소는 여러 프로젝트가 공유하므로, 어떤 프로젝트의 태스크인지 구분이 필요하다.

- 형식: 소괄호로 감싼 프로젝트명 — `(프로젝트명)`
- 프로젝트명은 CWD의 Git 저장소 루트 폴더명을 사용한다
- 예: CWD가 `~/Workspace/hoho-hr`이면 scope는 `(hoho-hr)`
- 최종 경로: `.wiki/tasks/(hoho-hr)/`

위키 저장소를 하나의 프로젝트에서만 사용하더라도 scope 폴더 구조는 유지한다. 추후 다른 프로젝트가 같은 위키를 공유할 때 구조 변경 없이 확장할 수 있다.

## 태스크 파일

작업 지시서 역할을 하는 마크다운 파일이다. 작업의 의도와 참고 자료만 기록하며, 구현 결과는 Git 커밋과 태그에 위임한다.

### 파일명 형식

```
vx.x.x-(feat|fix|refactor|chore|docs)-작업명.md
```

### 템플릿

```markdown
# v0.2.0-feat-디자인시스템구축

> 상태: todo

## 지시 내용

- (작업 요청 사항)

## 참고 문서

- (관련 문서 경로/링크)
```

### 상태

`todo` → `in-progress` → `done`

blockquote(`>`) 형식으로 파일 상단에 기록한다.

## 버전 결정

태스크 파일명의 버전은 scope 폴더의 기존 파일들에서 가장 높은 버전을 기준으로 결정한다.

- 기존 파일이 없으면 `v0.1.0`부터 시작
- `feat` → minor 버전 증가 (v0.1.0 → v0.2.0)
- `fix`, `refactor`, `chore`, `docs` → patch 버전 증가 (v0.1.0 → v0.1.1)

## 타입

| 타입 | 용도 | 브랜치 접두사 |
|------|------|---------------|
| `feat` | 새 기능 추가 | `feat/` |
| `fix` | 버그 수정 | `fix/` |
| `refactor` | 동작 변경 없는 코드 개선 | `refactor/` |
| `chore` | 빌드, 설정, 의존성 등 유지보수 | `chore/` |
| `docs` | 문서 작업 | `docs/` |

## Git 브랜치 전략

| 브랜치 | 역할 |
|--------|------|
| `main` | 안정 배포 브랜치 |
| `develop` | 개발 통합 브랜치 |
| `{타입}/xxx` | 작업 브랜치 (develop에서 분기) |

- 브랜치명은 영문으로 작성한다
- 태스크 파일의 작업명을 영문으로 변환하여 사용한다
- 예: `v0.2.0-feat-디자인시스템구축` → 브랜치 `feat/design-system`
