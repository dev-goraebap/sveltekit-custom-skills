# 제품 요구사항 문서(PRD) 생성

## 목표

초기 사용자 프롬프트를 바탕으로 Markdown 형식의 상세한 제품 요구사항 문서(PRD)를 작성하도록 AI 어시스턴트를 안내한다. PRD는 명확하고 실행 가능해야 하며, 주니어 개발자가 기능을 이해하고 구현할 수 있는 수준으로 작성되어야 한다.

## 프로세스

1. **초기 프롬프트 수신:** 사용자가 새로운 기능 또는 요구사항에 대한 간략한 설명을 제공한다.
2. **명확화 질문:** PRD를 작성하기 전에, AI는 명확한 PRD 작성에 필요한 핵심 질문만 반드시 먼저 해야 한다. 질문은 3~5개의 중요한 불명확 사항으로 제한한다. 목표는 기능의 "무엇"과 "왜"를 이해하는 것이며, "어떻게"는 개발자가 결정한다. 사용자가 쉽게 답할 수 있도록 번호/문자 목록으로 선택지를 제공한다.
3. **PRD 생성:** 초기 프롬프트와 명확화 질문에 대한 답변을 바탕으로 아래 구조에 따라 PRD를 생성한다.
4. **PRD 저장:** 생성된 문서를 `/blueprint` 디렉토리 안에 `prd-[기능명].md` 형식으로 저장한다.

## 명확화 질문 (가이드라인)

PRD 작성에 필요한 가장 핵심적인 질문만 한다. 초기 프롬프트에서 모호하거나 필수 맥락이 빠진 부분에 집중한다. 주로 명확화가 필요한 영역은 다음과 같다.

- **문제/목표:** 불분명한 경우 — "이 기능이 사용자의 어떤 문제를 해결하는가?"
- **핵심 기능:** 모호한 경우 — "사용자가 수행할 수 있어야 하는 핵심 동작은 무엇인가?"
- **범위/경계:** 광범위한 경우 — "이 기능이 하지 *말아야* 할 것이 있는가?"
- **성공 기준:** 명시되지 않은 경우 — "이 기능이 성공적으로 구현되었음을 어떻게 알 수 있는가?"

**중요:** 초기 프롬프트에서 합리적으로 추론 가능한 사항은 질문하지 않는다. PRD의 명확성에 큰 영향을 미치는 질문을 우선시한다.

### 형식 요구사항

- **질문은 숫자로 번호를 매긴다** (1, 2, 3, ...)
- **각 질문의 선택지는 A, B, C, D, ...로 표기한다**
- 사용자가 "1A, 2C, 3B"와 같이 간단히 답할 수 있도록 한다

### 예시 형식

```
1. 이 기능의 주요 목표는 무엇인가?
   A. 사용자 온보딩 경험 개선
   B. 사용자 리텐션 증가
   C. 지원 부담 감소
   D. 추가 수익 창출

2. 이 기능의 대상 사용자는 누구인가?
   A. 신규 사용자만
   B. 기존 사용자만
   C. 전체 사용자
   D. 관리자 사용자만

3. 이 기능의 예상 일정은?
   A. 긴급 (1~2주)
   B. 높은 우선순위 (3~4주)
   C. 일반 (1~2개월)
   D. 추후 검토 (3개월 이상)
```

## PRD 구조

생성되는 PRD는 다음 섹션을 포함해야 한다.

1. **소개/개요:** 기능과 해결하려는 문제를 간략히 설명한다. 목표를 명시한다.
2. **목표:** 이 기능의 구체적이고 측정 가능한 목적을 나열한다.
3. **사용자 스토리:** 기능 사용 방식과 이점을 설명하는 사용자 관점의 서술을 작성한다. 각 항목에 `US-01`, `US-02`, ... 형식으로 ID를 부여한다.
4. **기능 요구사항:** 기능이 반드시 갖춰야 할 구체적인 항목을 나열한다. 명확하고 간결한 언어를 사용한다 (예: "시스템은 사용자가 프로필 사진을 업로드할 수 있어야 한다."). 각 항목에 `FR-01`, `FR-02`, ... 형식으로 ID를 부여한다.
5. **비목표 (범위 외):** 범위 관리를 위해 이 기능이 포함하지 *않을* 사항을 명확히 명시한다.
6. **디자인 고려사항 (선택):** 목업 링크, UI/UX 요구사항, 또는 관련 컴포넌트/스타일 등을 기술한다.
7. **기술 고려사항 (선택):** 알려진 기술적 제약, 의존성, 또는 제안 사항을 언급한다 (예: "기존 Auth 모듈과 통합되어야 한다").
8. **성공 지표:** 이 기능의 성공을 어떻게 측정할 것인가? (예: "사용자 참여율 10% 증가", "X 관련 지원 티켓 감소").
9. **미결 질문:** 남아 있는 질문이나 추가 명확화가 필요한 영역을 나열한다.

## 대상 독자

PRD의 주요 독자는 **주니어 개발자**임을 전제한다. 따라서 요구사항은 명시적이고 모호함이 없어야 하며, 가능한 한 전문 용어 사용을 피한다. 기능의 목적과 핵심 로직을 이해할 수 있을 만큼 충분한 수준의 상세함을 제공한다.

## 출력

- **형식:** Markdown (`.md`)
- **위치:** `/blueprint/`
- **파일명:** `prd-[기능명].md`

## 최종 지시사항

1. PRD 구현을 바로 시작하지 않는다
2. 반드시 사용자에게 명확화 질문을 먼저 한다
3. 사용자의 답변을 반영하여 PRD를 개선한다
